# Torch oxDNA2 Notes

## Scope

This repository is still the upstream `oxDNA` codebase first. The PyTorch layer lives under `subho/` and is focused on **oxDNA2 DNA energy evaluation**, not on oxDNA1 compatibility.

Current defaults in `subho/oxdna2_pytorch.py`:

* `interaction model`: oxDNA2
* `grooving=True`: major/minor-groove backbone geometry is enabled
* `use_average_seq=False`: sequence-dependent parameters are used by default
* `seq_dep_file`: defaults to `oxDNA2_sequence_dependent_parameters.txt` at the repo root

## High-level layout

Core oxDNA pieces:

* `src/`: C++ simulation and observable implementation
* `analysis/src/oxDNA_analysis_tools/`: Python analysis tools
* `src/Observables/PairEnergy.cpp`: oxDNA CPU pair-energy observable
* `analysis/src/oxDNA_analysis_tools/output_bonds.py`: CLI/API wrapper around `pair_energy`

PyTorch layer:

* `subho/oxdna2_pytorch.py`: main Torch implementation
* `subho/example_torch_usage.py`: example system evaluation on an oxDNA `.top` + `.dat`
* `subho/test_against_reference.py`: legacy comparison against stored JSON overlays
* `subho/compare_single_frame_to_oat.py`: current one-frame oxDNA CPU vs Torch comparison
* `subho/compat/zstandard.py`: narrow compatibility stub for local plain-text fallback runs

## How the Torch implementation works

### Input convention

`oxDNA2Energy` expects:

* `positions`: `[N, 3]` center-of-mass coordinates
* `orientations`: `[N, 3, 3]` rotation matrices with columns `(a1, a2, a3)`
* `base_types`: `[N]` with `0=A, 1=C, 2=G, 3=T`
* `n3_neighbors`, `n5_neighbors`: `[N]` topology neighbor indices

The `.dat` reader reconstructs `a2` from `a3 x a1`, matching oxDNA's orientation convention.

### Implemented energy terms

`subho/oxdna2_pytorch.py` implements the oxDNA2 terms:

* `FENE`
* `BEXC`
* `STCK`
* `NEXC`
* `HB`
* `CRSTCK`
* `CXSTCK`
* `DH`

### Full-system GPU path

The system path is `oxDNA2Energy.compute_system_energies(...)`.

What it does:

1. Builds all `i < j` particle pairs on-device.
2. Classifies pairs as bonded or non-bonded from topology.
3. Evaluates all energy terms in vectorized Torch code.
4. Returns both pairwise and per-nucleotide views.

Returned data:

* `pair_terms`: per-pair term arrays
* `pair_total`: total per-pair energy
* `term_sums`: pairwise system sums for each term
* `total_sum`: total pairwise system energy
* `per_nucleotide_terms`: split per-nucleotide energies
* `per_nucleotide_total`: split per-nucleotide total energies

`per_nucleotide_*` uses **split accounting**:

* each pair contribution is divided equally between its two endpoints
* therefore `sum(per_nucleotide_total) == total_sum`

This split is done by `_split_pair_energies(...)`.

### Backbone geometry

Backbone positions are not hard-coded as the oxDNA1 site. The implementation uses `_backbone_site(...)`:

* `grooving=False`: oxDNA-style `POS_BACK`
* `grooving=True`: oxDNA2-style major/minor-groove offset using `POS_MM_BACK1` and `POS_MM_BACK2`

This matters directly for:

* `FENE`
* excluded-volume terms involving backbone sites
* Debye-Huckel

### Sequence dependence

When `use_average_seq=False`, the model loads:

* `STCK_FACT_EPS`
* all `STCK_X_Y` entries
* `HYDR_A_T` / `HYDR_T_A`
* `HYDR_G_C` / `HYDR_C_G`

from `oxDNA2_sequence_dependent_parameters.txt`.

This mirrors the oxDNA C++ logic for DNA2 sequence-dependent parameters instead of using average-sequence constants.

## Ground truth generation

The current validation path does **not** treat the old JSON files in `subho/example/` as the authoritative source for oxDNA2.

The authoritative ground truth is generated by oxDNA CPU:

1. `analysis/src/oxDNA_analysis_tools/output_bonds.py` configures the `pair_energy` observable.
2. `src/Observables/PairEnergy.cpp` calls the C++ interaction code term-by-term for each interacting pair.
3. `output_bonds -v` converts those pair energies into per-nucleotide arrays.

The important subtlety is the accounting convention.

## Accounting convention

`output_bonds -v` writes **raw oxView overlays**:

* for each pair `(i, j)`, the full pair energy is added to nucleotide `i`
* the same full pair energy is also added to nucleotide `j`

So the raw per-nucleotide overlays are **double-counted** with respect to pairwise system energy.

That means:

* `sum(output_bonds_overlay) = 2 * pairwise_system_energy`

The Torch full-system path uses split accounting instead:

* `i += E(i, j) / 2`
* `j += E(i, j) / 2`

So a fair comparison must convert one side to the other's convention.

The script `subho/compare_single_frame_to_oat.py` does this explicitly:

* loads raw `output_bonds -v` JSON arrays
* divides them by `2`
* compares those split arrays against `per_nucleotide_terms` and `per_nucleotide_total`
* also compares pairwise system sums against `term_sums` and `total_sum`

## Single-frame comparison script

`subho/compare_single_frame_to_oat.py` is the preferred validation entry point.

Example:

```bash
python subho/compare_single_frame_to_oat.py \
    --top subho/example/dsdna8.top \
    --frame subho/example/init.dat \
    --temperature 20C \
    --salt 0.5
```

What it does:

1. Loads the `.top` and one single-frame `.dat`.
2. Evaluates the Torch model on the active Torch device.
3. Generates a temporary oxDNA input file with:
   * `interaction_type = DNA2`
   * `use_average_seq = false` by default
   * the chosen temperature and salt
4. Runs `output_bonds` on the same frame.
5. Converts raw per-nucleotide overlays to split accounting.
6. Reports:
   * pair-sum differences
   * RMSE
   * max absolute error
   * mean absolute error

### Runner resolution

The script prefers:

* `oat output_bonds`

If `oat` is not installed, it falls back to the repo-local analysis build:

* `/usr/bin/python3`
* `analysis/src/oxDNA_analysis_tools/output_bonds.py`
* local built `oxpy` from `build/.../oxpy/python`
* local built analysis extension from `analysis/build/...`

The fallback also prepends `subho/compat/` so plain `.dat` files can be processed even if the real `zstandard` package is not installed. That stub is only acceptable for uncompressed trajectories.

## Why the old reference comparison was misleading

The old `subho/test_against_reference.py` workflow compares against stored JSON overlays in `subho/example/output_*.json`.

Those files were generated under older settings and are useful only if the Torch model is forced into the same configuration. In particular:

* oxDNA input defaults to `interaction_type = DNA` unless specified otherwise
* oxDNA defaults to `use_average_seq = true`
* raw `output_bonds` overlays are double-counted

So those stored JSONs are not the right source of truth for strict oxDNA2 sequence-dependent validation.

## Device support

Torch evaluation supports:

* `cpu`
* `cuda`
* `mps`

The full-system path is written to stay on-device for the Torch computation itself. The oxDNA CPU ground-truth side remains CPU because it comes from the oxDNA C++ observable stack.

## Practical rule

If the question is "does the current Torch oxDNA2 implementation match oxDNA2 CPU on this frame?", use:

* `subho/compare_single_frame_to_oat.py`

If the question is "how do I call the Torch model inside Python?", use:

* `subho/example_torch_usage.py`

If the question is "how does oxDNA itself compute the ground-truth pair energies?", read:

* `src/Observables/PairEnergy.cpp`
* `analysis/src/oxDNA_analysis_tools/output_bonds.py`
